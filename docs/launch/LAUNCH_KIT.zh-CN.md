# Kimi Partner 首发宣传包

## 统一定位

**核心句：**设计师定方向，模型加速落地。

**一句话介绍：**Kimi Partner 把 Kimi 的前端实现能力接入 Codex 工作流：设计师负责目标、规范与最终判断，Kimi 完成限定范围的前端实现，Codex 负责边界控制、代码审查、测试与浏览器验收。

**辅助句：**让 Kimi 发挥前端表现力，让 Codex 守住工程质量。

Kimi Partner 是一个独立社区项目，与 OpenAI、Moonshot AI 或 Kimi 均无隶属、合作或官方背书关系。

## V2EX / 掘金

### 标题

我做了一个 Codex 插件，把 Kimi 接进受约束的前端实现流程

### 问题

我是 Jevon，一名 UI/UX 设计师。在使用 AI 落地界面时，我常遇到一个很具体的问题：模型可以很快写出页面，但真实项目并不只需要“生成结果”。设计目标由谁定义？哪些文件允许修改？变更是否越界？代码、测试和浏览器表现由谁检查？这些问题如果没有明确答案，速度越快，返工也可能越多。

我喜欢 Kimi 在不少前端界面任务里的视觉判断和成品感，但这是一种使用体验，不是“所有任务都优于其他模型”的性能结论。我也希望保留 Codex 在任务拆解、工程控制和独立验收上的角色，因此做了 Kimi Partner。

### 设计决策

Kimi Partner 的核心不是模型替换，而是职责分离：

- 设计师定义问题、产品目标、视觉方向、设计规范与最终验收标准；
- Kimi 只负责经过明确选择、限定范围的前端实现；
- Codex 负责限定边界、检查代码与 Git diff、运行测试并完成浏览器验收。

一句话概括就是：**设计师定方向，模型加速落地。**

### 工作流

1. 设计师给出页面目标、允许修改范围和视觉验收标准。
2. Codex 理解项目并把可修改文件写入任务边界；只有用户明确选择或批准后，才启动 Kimi Partner。
3. Kimi 在限定范围内实现前端任务，任务可异步查询，并可在同一 Kimi 会话中接收返修意见。
4. Codex 对照 Git 基线检查实际变更，独立审查 diff、运行测试并做浏览器验收。
5. 设计师完成最终视觉与体验判断；不满足标准时，再带着具体证据返修。

### 护栏

插件会规范化允许路径，检查符号链接逃逸、范围外写入、项目外绝对路径、未授权依赖安装和危险 Git 命令；每轮结束后还会核对 Git HEAD 与实际改动。如果出现越界修改或 HEAD 变化，任务会失败，而不是把结果当成正常完成。

这些是**应用层护栏，不是操作系统沙箱**。Kimi 的完成总结也只是检查线索，不能代替 Codex 的独立代码审查、测试和浏览器验收。

### 安装

前置条件：macOS、Node.js 22+、已安装并登录 Kimi Code CLI，以及支持 `codex plugin` 命令的 Codex。目标项目需要位于 Git 工作区内。

```bash
git clone https://github.com/jevonhou/kimi-partner.git ~/plugins/kimi-partner
cd ~/plugins/kimi-partner
npm ci
npm run verify
```

接着按照仓库 README，把本地插件注册到个人 Codex marketplace，执行 `codex plugin add kimi-partner@personal`，然后新开一个 Codex 任务加载 Skill 和四个 MCP 工具。若个人 marketplace 使用了其他名称，请以本机配置为准。

第一次建议从小范围任务开始，例如：

> 让 Kimi 实现这个项目的按钮交互状态，只允许修改 `src/components/Button.tsx` 和 `src/styles/button.css`。完成后你自己检查 diff 并验收。

### 项目状态

项目当前为 `0.1.x` 早期版本，适合本地试用并配合严格代码审查。当前只正式验证了 macOS；Windows 和 Linux 的进程组行为暂未承诺。源码与完整安装说明：<https://github.com/jevonhou/kimi-partner>

## 常见问题

### 是不是替代设计师？

不是。设计师负责问题定义、产品目标、视觉方向、设计规范与最终判断。Kimi Partner 只把限定范围的前端实现交给 Kimi，再由 Codex 做工程检查和验收；它不能代替完整的产品设计工作。

### 为什么不是直接用 Kimi？

你当然可以直接使用 Kimi。这个插件解决的是另一类需求：当工作已经在 Codex 和真实 Git 项目中进行时，把一项明确批准的前端任务交给 Kimi，同时保留路径边界、异步状态、Git 对账、代码审查、测试与浏览器验收闭环。

### 会不会自动改整个仓库？

不会自动委派，也不应该把整个仓库默认开放给任务。Codex 需要为每次任务声明允许修改的路径；插件会检查范围外改动和 Git HEAD 变化，发现越界时任务直接失败。不过这些是应用层护栏，不是系统沙箱，所以 Codex 仍必须检查实际 diff。

### 是不是官方插件？

不是。Kimi Partner 是 Jevon Hou 维护的独立社区项目，与 OpenAI、Moonshot AI 或 Kimi 均无隶属、合作或官方背书关系。

### 支持哪些系统？

当前正式验证 macOS，并要求 Node.js 22+、已安装且登录的 Kimi Code CLI、支持插件命令的 Codex，以及位于 Git 工作区内的目标项目。Windows 和 Linux 的进程组行为尚未正式验证，因此目前不做兼容承诺。

### Kimi 完成后可以直接合并吗？

不建议。Kimi 的总结只是检查线索。Codex 需要独立核对 Git diff、代码、测试和浏览器结果，设计师还要完成最终视觉与体验判断；全部满足后，才进入项目原有的提交或合并流程。
