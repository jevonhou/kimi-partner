## Highlights

- Made one five-minute `wait_kimi_task` call the default active-task waiting behavior.
- Increased the fallback polling suggestion from 20 seconds to 60 seconds.
- Updated the Skill to request `wait_ms: 300000` explicitly and avoid narrating unchanged state.

## Why this release

Frontend generation can take tens of minutes. The previous 45-second default already removed high-frequency polling, but a 30-minute task could still require about 40 Codex tool cycles. With the five-minute default, the same task needs about six waits while active responses remain compact.

## Real host verification

- A 120-second `wait_kimi_task` call completed in 120.170 seconds.
- A 300-second `wait_kimi_task` call completed in 300.267 seconds.
- Neither call was terminated at 60 seconds.
- Both calls returned only the compact active-state fields.
- The probes used a synthetic persistent task and did not invoke Kimi or modify a project.

## Compatibility and safety

- The tool names, persisted task format, 5-minute maximum, terminal response, and Git change receipt remain compatible with `v0.1.2`.
- `get_kimi_task` remains available for immediate snapshots and backward-compatible short waits.
- Codex still owns scope definition, Git review, testing, browser acceptance, and final verification.

## Verification

- 47 / 47 automated tests pass.
- The bundled five-tool MCP server passes isolated-distribution and syntax checks.
- Dependency audit reports 0 known vulnerabilities.
