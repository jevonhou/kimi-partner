# Kimi Partner 首发宣传包设计说明

**日期：** 2026-07-18  
**状态：** 已确认方向，待用户复核书面规格

## 目标

为 Kimi Partner 制作一套可直接用于开源项目首发的中文宣传包。宣传要清楚解释插件的价值，同时避免把 Kimi 描述成“一键替代设计师”的工具。

首发的核心目标不是制造模型对立，而是吸引正在使用 Codex、对 Kimi 前端表现力感兴趣的设计师和开发者，让他们理解 Kimi Partner 的真实协作方式并访问 GitHub 仓库。

## 核心定位

主叙事固定为：

> 设计师定方向，模型加速落地。  
> Kimi 负责前端实现，Codex 负责工程控制与验收。

设计师始终是工作流的主导者，负责问题定义、产品目标、视觉方向、设计规范与最终判断。Kimi 和 Codex 是能力不同的执行搭档，而不是替代设计判断的自动化按钮。

## 受众

首发优先面向：

1. 使用 Codex 完成本地项目、但希望尝试 Kimi 前端实现能力的设计师与开发者。
2. 希望提高设计落地效率，同时保留修改边界、代码审查与验收闭环的 UI/UX 设计师。
3. 关注多模型协作、Codex Plugin 与 MCP 工作流的开源用户。

不把“老板用 AI 减少设计岗位”作为目标场景，也不使用降本、裁员或替代人力的表达。

## 视觉方向

采用已经确认的“编辑式极简”方向：

- 以白色和浅灰为背景，保持开源工具的可信、克制与清晰感。
- 使用近黑色主文字，以现有 Logo 的蓝色和紫色作为连接色。
- 使用宽松留白、清晰网格和简洁的无衬线字体。
- Logo 保持独立，不修改几何结构，也不模仿 OpenAI、Moonshot AI 或 Kimi 的官方标志。
- 画面明确展示三方分工：设计师负责目标与判断，Kimi 负责前端实现，Codex 负责审查与验收。

主封面标题为“设计师定方向，模型加速落地”。辅助文案为“让 Kimi 发挥前端表现力，让 Codex 守住工程质量”。

## 30 秒演示结构

演示必须展示真实、可控的协作流程：

1. **0–5 秒：设计目标**  
   设计师给出页面目标、允许修改范围和视觉验收标准。
2. **5–10 秒：任务交接**  
   Codex 明确限定可修改文件，并启动 Kimi Partner。
3. **10–19 秒：Kimi 实现**  
   展示任务运行和完成后的真实界面结果，避免使用“魔法”“一键生成”等表达。
4. **19–26 秒：Codex 验收**  
   展示 Git diff、测试和浏览器检查。
5. **26–30 秒：设计师判断与行动入口**  
   明确最终视觉判断仍由设计师完成，并显示 GitHub 地址。

演示素材不能包含私人项目路径、令牌、账号信息或无法公开的客户内容。

## 表达边界

所有宣传内容必须遵循以下约束：

- 不宣传“设计师没用了”“一键替代设计师”或类似结论。
- 不把视觉润色等同于完整的产品设计工作。
- 不声称 Kimi 在所有前端任务上客观优于其他模型。
- 可以表述“很多开发者喜欢 Kimi 的前端视觉判断与成品感”，但不把主观偏好写成未经验证的性能结论。
- 不承诺操作系统沙箱级安全；继续准确说明插件提供的是应用层护栏。
- 保留项目与 OpenAI、Moonshot AI 和 Kimi 无官方隶属或背书关系的声明。

## 验收标准

宣传包完成时需要满足：

1. 两张封面尺寸准确、文字无错字、基线和网格对齐清晰。
2. 缩小到移动端预览尺寸后，标题和三方分工仍然可读。
3. 各平台文案使用统一定位，但长度和语气符合平台特点。
4. 演示脚本所有步骤都能由当前插件真实支持，不虚构能力。
5. README 演示内容不阻碍现有安装与安全说明的阅读。
6. 新增文件通过敏感信息与格式检查，仓库原有验证继续通过。
